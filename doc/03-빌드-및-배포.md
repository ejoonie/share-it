# 03. 빌드 및 배포

> 빌드부터 스토어 업로드·심사 제출까지 자동화한 절차입니다.
> 환경 설정은 `fastlane/.env` 한 파일로 관리하며, fastlane 실행 시 자동으로 로드됩니다.

## 명령어 요약

```bash
# 버전 올리고 (pubspec.yaml) 아래 한 줄이면 양쪽 스토어 업로드 + 심사 제출까지 완료
scripts/release.sh all
```

실행하면 릴리즈 노트 입력 프롬프트가 뜹니다:

```
📝  릴리즈 노트를 입력하세요 (스토어에 공개됩니다).
    여러 줄 입력 가능 — 완료하면 빈 줄에서 Enter:

회원탈퇴 기능 추가
안정성 개선
(빈 줄 Enter)
→ 빌드 → 업로드 → 심사 제출 자동 진행
```

`RELEASE_NOTES` 환경변수가 이미 있으면 프롬프트를 건너뜁니다:

```bash
RELEASE_NOTES="Bug fixes and stability improvements" scripts/release.sh all
```

## 1회 준비: `fastlane/.env`

fastlane은 dotenv를 내장하고 있어 `fastlane/.env` 파일을 실행 때마다 자동으로 읽습니다.
`export`나 `source`가 필요 없습니다.

```bash
bundle install
flutter pub get
cp fastlane/.env.example fastlane/.env   # 템플릿 복사 후 경로를 자기 머신에 맞게 수정
```

`.env`는 머신별 절대 경로가 들어가므로 **gitignore 되어 있습니다** (커밋 금지).
반면 자격증명 파일 자체는 관리 편의를 위해 저장소로 공유합니다.

| 자격증명 | 저장소 내 위치                              |
|---|---------------------------------------|
| Google Play 서비스 계정 JSON | `scripts/xxx-dfb08-7097ea19ea6e.json` |
| App Store Connect API 키 (`.p8`) | `AuthKey_HFM6VVW6Y5.p8` (프로젝트 루트)     |

### 현재 팀 표준 `.env` 구성

```bash
# ── Android / Google Play ──────────────────────────────
SUPPLY_JSON_KEY=/절대경로/scripts/xxx-dfb08-7097ea19ea6e.json
ANDROID_PACKAGE_NAME=com.sharablepiggy.app
PLAY_TRACK=production
PLAY_RELEASE_STATUS=completed        # 심사 통과 시 즉시 전체 공개

# ── iOS / App Store Connect ────────────────────────────
# Key ID/Issuer ID: https://appstoreconnect.apple.com/access/integrations/api
APP_STORE_CONNECT_API_KEY_KEY_ID=HFM6VVW6Y5
APP_STORE_CONNECT_API_KEY_ISSUER_ID=<Issuer ID>
APP_STORE_CONNECT_API_KEY_KEY_FILEPATH=/절대경로/AuthKey_HFM6VVW6Y5.p8
APPLE_DEVELOPER_TEAM_ID=WF4H2BG68T
APP_STORE_CONNECT_TEAM_ID=WF4H2BG68T
IOS_SKIP_METADATA=false              # 릴리즈 노트를 함께 업로드
IOS_AUTOMATIC_RELEASE=true           # 심사 승인 시 자동 출시

# ── 공통 ───────────────────────────────────────────────
RELEASE_NOTES=Automated production release build.
FLUTTER_FLAVOR=production
FLUTTER_TARGET=lib/main_production.dart
SUBMIT_FOR_REVIEW=true               # iOS 심사 자동 제출
```

> `APPLE_ID`(Apple 계정 이메일)는 API 키 인증을 쓰므로 필요 없습니다.

## 버전 관리 (`pubspec.yaml`)

- **위치**: 프로젝트 루트 `pubspec.yaml`의 `version` 필드
- **형식**: `버전+버전코드` 예: `1.1.3+135`

**릴리스 전 반드시 둘 다 올려야 합니다:**

- **버전**(`1.1.3`)이 App Store에 이미 출시된 버전과 같으면 iOS 심사 제출 불가
- **버전코드**(`135`)가 Google Play에 한 번이라도 올라간 값이면 Android 업로드 거부
  (versionCode는 트랙 구분 없이 전역 유니크)

Flutter가 `version` 값을 Android `versionName/versionCode`, iOS `MARKETING_VERSION/CURRENT_PROJECT_VERSION`으로 전달합니다.

## 자동화 구성

| 파일 | 역할 |
|---|---|
| `Gemfile` | fastlane Ruby gem 버전 고정 |
| `fastlane/.env` | 로컬 환경 설정 (자동 로드, gitignore) |
| `fastlane/.env.example` | `.env` 템플릿 (커밋 대상) |
| `fastlane/Appfile` | iOS bundle id, Android package name, 팀/인증 환경변수 연결 |
| `fastlane/Fastfile` | 빌드, TestFlight, App Store Connect, Google Play 업로드 lane |
| `scripts/release.sh` | 진입점. 릴리즈 노트를 콘솔에서 입력받고 이전 빌드 산출물 자동 삭제 후 실행 |

Fastfile에 반영된 주의사항:

- API 키 인증으로는 인앱결제 precheck가 불가능해 `precheck_include_in_app_purchases: false`로 제외
- Google Play 릴리즈 노트는 supply에 파라미터로 넘길 수 없어
  `fastlane/metadata/android/<언어>/changelogs/default.txt` 파일로 자동 생성해 전달

## 빌드만 하기

스토어 업로드 없이 iOS IPA와 Android AAB를 모두 생성합니다.

```bash
scripts/release.sh build
```

**아티팩트 위치**

- iOS IPA: `build/ios/ipa/*.ipa`
- iOS Archive: `build/ios/archive/*.xcarchive`
- Android AAB: `build/app/outputs/bundle/productionRelease/*.aab`

## 플랫폼별 배포

```bash
scripts/release.sh ios        # iOS만
scripts/release.sh android    # Android만
scripts/release.sh all        # 둘 다 (iOS → Android 순차)
```

보수적으로 배포하고 싶을 때 일회성으로 덮어씁니다:

```bash
PLAY_TRACK=internal scripts/release.sh android          # internal 트랙
PLAY_RELEASE_STATUS=draft scripts/release.sh android    # 콘솔에서 수동 출시
SUBMIT_FOR_REVIEW=false scripts/release.sh ios          # 업로드만, 심사 제출 보류
```

> `all`은 iOS가 실패하면 Android를 시작하지 않습니다.

## 주요 환경변수

| 변수 | 팀 표준값 | 설명 |
|---|---|---|
| `RELEASE_NOTES` | `Automated production release build.` | 스토어에 공개 노출 — 배포 시 실제 내용으로 덮어쓸 것 |
| `SUBMIT_FOR_REVIEW` | `true` | iOS App Review 자동 제출 |
| `IOS_AUTOMATIC_RELEASE` | `true` | iOS 심사 승인 후 자동 출시 |
| `IOS_SKIP_METADATA` | `false` | iOS 메타데이터(릴리즈 노트 포함) 업로드 |
| `IOS_SKIP_BINARY_UPLOAD` | 미설정 | 바이너리 이미 올라간 상태에서 심사 제출만 재시도할 때 `true` |
| `PLAY_TRACK` | `production` | `internal` / `alpha` / `beta` / `production` |
| `PLAY_RELEASE_STATUS` | `completed` | `completed`=즉시 공개, `draft`=콘솔에서 수동 출시 |

## 트러블슈팅

**`Cannot find edit app store version... Retrying`**

1. `pubspec.yaml` 버전이 App Store 출시 버전과 동일 → 버전 올리기
2. IPA 경로 문제 → `latest_ipa` 헬퍼 확인

**`Version code XXX has already been used`**

`pubspec.yaml` 버전코드가 Play에 이미 존재 → 버전코드 올리기. 현재 트랙 확인:

```bash
bundle exec fastlane run google_play_track_version_codes \
  package_name:us.hapakristin.app track:production \
  json_key:scripts/hapaus-dfb08-7097ea19ea6e.json
```

**iOS 바이너리 업로드는 성공했는데 심사 제출에서 실패한 경우**

빌드·업로드 재실행 없이 제출만 재시도:

```bash
IOS_SKIP_BINARY_UPLOAD=true scripts/release.sh ios
```

## 릴리스 체크리스트

1. `pubspec.yaml`의 `version` 올리기 (버전·버전코드 모두)
2. 코드 생성이 필요하면 실행:
   ```bash
   dart run build_runner build --delete-conflicting-outputs
   ```
3. 기본 검증:
   ```bash
   flutter analyze --fatal-warnings
   flutter test
   ```
4. 배포:
   ```bash
   scripts/release.sh all
   ```
5. 확인: App Store Connect에서 **심사 대기(Waiting for Review)** 상태, Play Console production 트랙에 새 versionCode 확인
